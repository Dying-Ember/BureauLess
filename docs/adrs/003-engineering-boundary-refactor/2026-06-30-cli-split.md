# ADR-003.2: CLI Command Ownership Split

## Status

Accepted.

## RFC

- Source RFC: [`docs/rfcs/003-engineering-boundary-refactor.md`](../../rfcs/003-engineering-boundary-refactor.md)
- Related task: [`docs/tasks/engineering_boundary_refactor_tasklist.md`](../../tasks/engineering_boundary_refactor_tasklist.md)

## Context

`src/bureauless/cli/main.py` currently owns parser construction, command
dispatch, legacy DAG commands, mission demo setup, ledger and workflow runtime
commands, assignment/result import flows, decision artifacts, agent operations,
session execution, and metrics summarization.

That makes the CLI hard to change safely. It also encourages future CLI and API
flows to duplicate orchestration logic because there is no small command module
boundary where ownership is clear.

## Decision

1. Keep `src/bureauless/cli/main.py` as the public CLI entrypoint.
2. Move command registration and handlers into command-specific modules under
   `src/bureauless/cli/`.
3. Preserve all current command names, arguments, stdout/stderr behavior, and
   exit codes while splitting ownership.
4. Start with bounded command groups:
   - `legacy.py`: `validate`, `ready`, `prompt`, `record`, and `review`.
   - `mission.py`: `mission validate`, `mission golden-path`,
     `mission mutation-demo`, `mission live-demo`, and demo workspace helpers.
   - `runtime.py`: `workflow`, `ledger`, `gatekeeper`, and `artifact`.
   - `exchange.py`: `assignment`, `result`, and `decision`.
   - `agents.py`: `agent` commands.
   - `sessions.py`: `session` commands.
   - `metrics.py`: `metrics` commands.
5. Leave broader orchestration extraction to the application-services ADR.

## Consequences

- `main.py` becomes parser setup, command registration glue, and top-level
  error handling.
- Command modules can be tested and reviewed by ownership area.
- API callers should not import CLI command modules for business behavior; they
  should use future application services instead.
- The first split is allowed to duplicate a small amount of parser boilerplate
  if that keeps behavior compatibility obvious.

## Implementation

- `src/bureauless/cli/main.py`
- `src/bureauless/cli/legacy.py`
- `src/bureauless/cli/mission.py`
- `src/bureauless/cli/runtime.py`
- `src/bureauless/cli/exchange.py`
- `src/bureauless/cli/agents.py`
- `src/bureauless/cli/sessions.py`
- `src/bureauless/cli/metrics.py`
- `tests/test_harness.py`
